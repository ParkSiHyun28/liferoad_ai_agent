# 시뮬레이터 모듈 데이터 계약 (CONTRACT)

각 에이전트 담당자는 자기 파트 JS 모듈 파일 1개를 작성한다. 리더가 셸 HTML에 통합한다.
이 계약을 어기면 통합이 깨진다. 형식을 글자 그대로 지켜라.

---

## 1. 큰 그림

서비스명: **My LifeRoad**. 외국인 입국→출국 전 과정을 조율하는 멀티에이전트.
오케스트레이터 1 + 에이전트 3 (서류 `docs`, 자산 `asset`, 사기탐지 `fraud`).
두 페르소나의 생애주기 4단계(입국/정착/체류/귀국or전환)에서 에이전트들이 서로 메시지를 주고받으며 능동적으로 업무를 처리하는 모습을 타임라인으로 재생한다.

핵심 연출 원칙 (노션 자산 페이지 3원리 반영):
1. 능동성 — 사용자가 묻기 전에 에이전트가 먼저 마감/위험을 감지해 행동 시작.
2. 개인화 — 페르소나의 실제 숫자(소득, 납부 월수, 잔고)로 답한다.
3. 대리 처리 — 다음 행동(서류 자동작성, 송금 연결, 홀딩)까지 잇는다.

## 2. 페르소나 2명 (공통 참조)

### 응웬 반 민 (NGUYEN VAN MINH) — 베트남 E-9 근로자
- 코드: `minh`, 국적 베트남(vn), 비자 E-9
- 체류: 2022.3 입국 → 2027.1 출국 예정. 현재 입국 4년차, 출국 약 1년 전.
- 경남 김해 제조업, 월 실수령 약 270만 원. 한국어 생활회화 수준.
- 핵심 문제: 받을 권리 있는 돈(국민연금 반환일시금, 출국만기보험)을 모름. 청구 마감 존재 자체를 모름.

### 수만 라이 (SUMAN RAI) — 네팔 D-2 유학생
- 코드: `suman`, 국적 네팔(np), 비자 D-2
- 체류: 2023.3 입국 → 2027.2 졸업 예정. 현재 3년차, 졸업/진로 전.
- 서울 사립대, 잔고증명용 예치금 약 2,000만 원이 묶임. 주 20시간 알바.
- 핵심 문제: 연금 함정(상호주의 제외국 네팔은 귀국 시 반환 비대상). Thin Filer(신용이력 얇음).

## 3. 4단계 (phase) 공통 정의

| phase 키 | minh 라벨 | suman 라벨 |
|---|---|---|
| `arrival` | 입국과 첫 정착 | 입국과 첫 정착 |
| `settle` | 정착 안정화 | 정착 안정화 |
| `expand` | 체류 확장 | 체류 확장 |
| `exit` | 귀국과 사후 | 졸업과 전환 |

## 4. 모듈 파일 형식 (글자 그대로 지킬 것)

파일명: 담당 파트별로 정해진 이름.
- 서류: `mod_docs.js`
- 자산: `mod_asset.js`
- 사기: `mod_fraud.js`

각 파일은 전역 객체 `window.LIFEROAD_MODULES`에 자기 파트를 push 하는 형태.
파일 맨 위에 가드 추가. 아래 골격을 그대로 복사해서 쓴다.

```js
// mod_<part>.js
(function () {
  window.LIFEROAD_MODULES = window.LIFEROAD_MODULES || {};

  window.LIFEROAD_MODULES["<part>"] = {
    id: "<part>",              // "docs" | "asset" | "fraud"
    name: "<에이전트 한글명>",   // 예: "서류 행정 에이전트"
    nameEn: "<English name>",  // 예: "Docs Agent"
    icon: "<이모지 1개>",       // 예: "📄"
    tagline: "<한 줄 역할 설명>", // 예: "비정형 서류를 정형 데이터로"
    // 이 에이전트가 가진 tool 목록 (노드 UI에 표시)
    tools: [
      { name: "register_baseline", desc: "정상 거래 분포 학습" },
      // ...
    ],
    // 시나리오 스텝: 페르소나별 배열
    steps: {
      minh: [ /* StepObject 배열 */ ],
      suman: [ /* StepObject 배열 */ ],
    },
  };
})();
```

### StepObject 형식 (시나리오의 1개 동작)

```js
{
  phase: "arrival",          // arrival|settle|expand|exit 중 하나
  t: 12,                     // 정렬용 정수 타임코드(0~99). 작을수록 먼저. 페르소나 전체에서 오름차순.
  actor: "docs",             // 이 스텝을 수행하는 주체. "orchestrator"|"docs"|"asset"|"fraud"|"user"
  to: "asset",               // 메시지 수신자. 위와 같은 토큰. 자기 작업이면 actor와 동일 가능. null 허용.
  kind: "message",           // "trigger"(능동 감지) | "message"(에이전트간 a2a) | "tool"(tool 실행) | "result"(사용자 카드 푸시) | "handoff"(다른 에이전트로 인계)
  title: "실명 표기 불일치 교정",   // 짧은 제목 (버스 타임라인 한 줄)
  detail: "외국인등록증과 근로계약서의 영문 성명 띄어쓰기 불일치 검출. NGUYEN VAN MINH vs NGUYENVANMINH",  // 한 문단 설명
  tool: "form_checker",       // kind가 "tool"일 때 실행한 tool 이름. 아니면 생략 가능.
  // kind가 "result"일 때만: 사용자 폰에 뜨는 카드
  card: {
    icon: "✅",
    head: "계좌 개설 준비 완료",
    body: "한도제한 해제 서류 패키지가 준비됐습니다. 전북은행 비대면 개설로 연결합니다.",
    metric: "처리 0.8초",      // 선택. 강조 수치/뱃지
  },
}
```

규칙:
- `t`는 페르소나 안에서 전 에이전트 통틀어 오름차순 재생된다. 리더가 모든 모듈의 steps를 합쳐 `t` 기준 정렬한다. 그러니 **단계별 t 구간을 지켜라**: arrival 10~24, settle 30~49, expand 55~74, exit 80~99. 같은 t 충돌 피하려 담당별 끝자리 배정: 서류=0,3,6,9 / 자산=1,4,7 / 사기=2,5,8.
- 에이전트 상호작용을 꼭 1회 이상 넣어라. 예: 자산이 연금 청구를 감지(`trigger`) → 서류에 `handoff`(서식 자동작성 요청) → 서류가 `tool` 실행 → 사기가 출국 직전 계좌양도 감시(`trigger`) → 사기가 `result` 카드. 담당자는 자기 `actor` 스텝만 작성하되 `to` 필드로 다른 에이전트를 가리켜 협업을 표현한다.
- detail에 노션 페이지의 실제 수치/제도명을 넣어라(반환일시금 3,294억, 출국만기보험 소멸시효 3년, 송금 5.15%, 잔고증명 2,000만, 예금담보 95%, 연체율 2.2~3% 등 자기 파트 관련).
- 텍스트는 자연스러운 한국어. 쉼표 남발 금지. 가운뎃점 문자(U+00B7 등) 절대 금지. "당신"/"너" 호칭 금지.
- 페르소나당 자기 파트 스텝 5~8개. 너무 적으면 빈약하고 많으면 늘어진다.

## 5. 각 담당 파트 핵심 내용 (노션 근거)

### 서류 (docs) — mod_docs.js
tool 예: form_checker(실명표기/날인 검출), cross_border_parser(해외 은행 거래내역 파싱), risk_scanner(임대차 전세사기 위험도), rule_validator(시간제취업/비자 가드레일), form_autofill(정부 PDF 자동 타이핑), closed_loop(출국 전 대출상환 가상계좌 매핑).
- minh: 입국 실명표기 교정→한도제한 해제 / 정착 베트남 은행 거래내역 파싱→금리 13.5%→6.5% / 체류 비자연장 D-90 선제 탐지→숙소제공확인서 / 귀국 반환일시금 지급신청서 Auto-fill + 대출 Closed-Loop.
- suman: 입국 임대차계약서 전세사기 위험 스캔(네팔어 경고) / 정착 시간제취업 확인서 대리작성 / 체류 잔고증명 검증 뱃지(신뢰도 97%) / 전환 E-7 고용사유서 직무연관성 교정 리포트.

### 자산 (asset) — mod_asset.js
6종: 예금, 대출, 송금, 연금, 보험, 신용(신규). 3원리(능동/개인화/대리처리). D-Day 마감 추적.
- minh: 입국 한도제한계좌 인지 + 송금 최저경로 비교(5.15%→저비용) / 정착 출국만기보험 적립 가시화(월 통상임금 8.3%) + 환치기 차단 / 체류 연금 "E-9는 수령대상" 정정 + 환율 적기 송금 / 귀국 반환일시금 예상액 개인 숫자 산출 + 보험금 소멸시효 D-Day + 서류로 handoff.
- suman: 입국 잔고요건 지키며 인출 가능범위 실시간 계산 / 정착 잔고증명 예금담보대출 95% 활용 + 월세와 통신비 신용데이터 축적 시작 / 체류 근로시간 추적 연금가입 시점 알림 + 네팔 상호주의 제외 손익구조 정직 안내 / 전환 신용프로필 결실 JB 대출과 카드 심사 활용.

### 사기 (fraud) — mod_fraud.js
fraud-guard tool 4개: register_baseline(segment×nationality 정상분포 학습), score_transaction(룰+Isolation Forest+LLM 맥락), detect_account_takeover(residency_end D-day 가중+일괄인출+신규디바이스), request_verification(모국어 본인확인 메시지 Claude 생성).
- minh: 입국 정상 baseline 등록(베트남 corridor) / 정착 베트남어 보이스피싱 본인확인 + 신규수취인 강화검증 / 체류 인출책 패턴(신규입금→즉시출금) 자동홀딩 / 귀국 D-30 계좌양도 가중치 ON→일괄인출+신규디바이스 홀딩(대포통장 626명과 12.8조 근거).
- suman: 입국 학교 공식계좌 화이트리스트 검증 / 정착 검찰과 경찰과 금감원 사칭 시나리오 패턴매칭 차단 / 체류 통장대여(비정상입금→즉시출금) 홀딩 + 위조알선 URL 차단 / 전환 졸업과 출국 임박 양도 가중치 ON + 비등록 환치기 송금 차단.

## 6. 작업 흐름

1. 자기 모듈 파일을 `/Users/parksihyun/공모전/output/sim_modules/mod_<part>.js`에 작성.
2. 작성 후 `node -e "require('./mod_<part>.js')"` 류로 문법 오류 없는지 자체 확인(window 없으면 globalThis 가드 통과되게 작성됨, 단순 문법체크는 node로 가능).
3. 자체검토 후 리더에게 SendMessage 1회로 보고(완료 파일 경로 + 페르소나별 스텝 수).
